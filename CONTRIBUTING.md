# Contributing to Uptimizr

Thanks for your interest in contributing! This guide covers the essentials. For deeper context,
read the [architecture overview](./docs/architecture/overview.md) and the
[ADRs](./docs/adr).

## Prerequisites

- **Node.js 22** (see `.nvmrc` — `nvm use`)
- **pnpm 11** (`corepack enable` or install per pnpm docs)
- **Docker** — _optional_. The OSS default store is a single DuckDB file, so local development
  needs **no Docker**. Docker is only required for the optional scale path (ClickHouse + Postgres).
  See [ADR 0020](./docs/adr/0020-open-core-storage-boundary.md).

## Setup

```bash
pnpm install
pnpm build               # build all packages once
cp .env.example .env     # adjust as needed
pnpm db:setup            # migrate the DuckDB store + seed a demo project + API key
```

`pnpm db:setup` prints a `PROJECT_ID` and a `utk_...` API key — paste them into `.env`
(`VITE_PROJECT_ID`, `VITE_API_KEY`). To reset the store, delete the `.duckdb` file (default
`./data/uptimizr.duckdb`) and re-run `pnpm db:setup`.

## Run the local stack (no Docker)

```bash
pnpm dev:stack           # collector + dashboard + playground together
```

Or start pieces individually: `pnpm dev:collector`, `pnpm dev:dashboard`, `pnpm dev:playground`.
Interact with the playground scene to generate events and confirm they appear in the dashboard.

> **Single-writer constraint:** DuckDB is an embedded, single-writer store — run only one
> collector against a given `.duckdb` file at a time.
>
> **Scale path:** ClickHouse + Postgres back the optional scale tier. Start them with
> `pnpm stack:up` (and `pnpm stack:down` to tear down) only if you opt into the scale store.

## Common commands

All tasks run through Turborepo from the repo root:

```bash
pnpm build       # build all packages
pnpm dev         # run dev tasks (watch)
pnpm lint        # eslint
pnpm typecheck   # tsc --noEmit per package
pnpm test        # unit/integration tests
pnpm test:e2e    # Playwright end-to-end tests (playground → collector → dashboard/replay)
pnpm format      # prettier --write
```

## Monorepo boundaries (important)

- **`oss/`** is Apache-2.0 and self-contained. Storage details stay behind the `@uptimizr/db`
  contracts so the store stays swappable. See
  [ADR 0004](./docs/adr/0004-monorepo-separation.md).
- Event shapes are defined **once** in `@uptimizr/schema` (Zod). Do not redefine event types
  elsewhere — import them.
- Keep business logic in framework-agnostic packages so backends (Fastify now, possibly Hono at
  the edge later) stay thin.

## Conventions

- **TypeScript strict** everywhere; no `any` without justification.
- **Validation at boundaries** — validate external input with Zod at the system edge, not deep
  inside.
- **Privacy first** — never introduce client-side persistent identifiers or collect PII by
  default (see [ADR 0003](./docs/adr/0003-privacy-model.md)).
- **Conventional Commits** for messages (e.g. `feat(schema): add camera_sample event`).
- **Document decisions** — significant choices get a new ADR (copy `docs/adr/template.md`).

## Pull requests

1. Branch from `main`.
2. Ensure `pnpm lint typecheck build test` pass locally.
3. Update or add docs/ADRs when behavior or decisions change. Any new feature, option, event, or
   endpoint must reach the public docs site (`oss/apps/docs`) and the integration/API reference
   (`docs/integration.md`) in the same PR — a feature isn't done until it's documented.
4. Major / user-visible features also need a **Playwright E2E** under `examples/playground/e2e/`
   that drives the real browser → SDK → collector → dashboard/replay round trip.
5. Keep PRs focused; describe the change and link related issues.

### AI-assisted contributions

We welcome AI-assisted work. **For transparency, if your contribution was authored primarily by an
AI agent — with little or no human review of the generated code — please say so in the PR
description** (e.g. "This PR was generated by an AI agent"). This helps reviewers calibrate the
level of scrutiny. You remain responsible for everything you submit: make sure it builds, passes
`pnpm lint typecheck build test`, and meets the conventions above.

## Adding things

- **A new event type?** Start in `@uptimizr/schema`, then thread it through `sdk-core`, the
  Babylon adapter, the collector, and replay. (A guided skill exists under `.github/skills`.)
- **A new 3D engine connector?** Mirror `@uptimizr/babylon` as a sibling package depending only
  on `@uptimizr/sdk-core` + `@uptimizr/schema`.

## Releases & changelog

Until the first published release, Uptimizr is pre-1.0 and the public packages are unversioned —
there is no changelog to maintain yet. The policy below takes effect when we start publishing
versioned packages to npm:

- **Versioning:** published `@uptimizr/*` packages follow [Semantic Versioning](https://semver.org).
  While pre-1.0, minor (`0.x`) bumps may carry breaking changes; we will call those out explicitly.
- **Changelog:** each publishable package keeps a `CHANGELOG.md` following
  [Keep a Changelog](https://keepachangelog.com). Entries are grouped under
  _Added / Changed / Fixed / Removed_ and derived from the Conventional Commit history since the
  last tag.
- **Conventional Commits drive it:** `feat:` → minor, `fix:` → patch, and a `!` or
  `BREAKING CHANGE:` footer → a breaking bump. Keep commit subjects accurate so release notes stay
  trustworthy.
- **Release notes:** tagged GitHub releases summarize user-facing changes and link the relevant
  ADRs.
